# X_AI Phase 5.5 RC1

更新日: 2026-07-17

このメモは、Phase 5.5 RC1 に向けた返信品質改善とリリース固定の要点をまとめたものです。Git 操作や実API呼び出しは行わず、確認できた事実だけを記録します。

## 変更の要点

### 返信品質改善

- 返信文を抽象的な説明調から、れいや本人が実際に返しそうな具体的なXリプへ寄せた。
- 「私たち」や企業広報調の抑止を強め、個人アカウントの自然な温度感へ寄せた。
- 元投稿への共感だけで終わらず、要件整理・確認・修正・更新フローのどこにAIを入れるかという一段深い視点を返せるようにした。
- 3案の役割を分けた。
  - A: 自然な補足
  - B: 実装経験からの具体視点
  - C: 短く鋭い別角度

### Context選択改善

- `selectedContextIds` を「広く関連しそうな文脈」ではなく、返信本文で実際に使う文脈へ寄せた。
- Web制作やAIの投稿で、営業・MEOなどの広い関連語に吸われにくいように補正した。
- `selectedContextIds` と `usedClaimEvidence` の整合を重視し、返信の中で実際に使う具体語と合わせる方針へ寄せた。
- 最大2件という上限を維持し、通常は1件で足りる場合は1件に抑える考え方を強めた。

### mock fixture 改善

- `web_and_ai` fixture を、抽象的な一般論ではなく具体的な工程名が入る返信例へ差し替えた。
- A/B/C の言い回し差をはっきりさせ、同じ意味の言い換えが並ぶ状態を避けた。
- fixture 側でも、実装・検証・確認・失敗などの具体語が残るように調整した。

### closure test 追加

- `runPhase37ClosureChecks.js` に、Web制作・AI 系の抽象語チェックを追加した。
- 禁止したい記事調表現が混入していないことを closure test で確認するようにした。
- mock fixture と prompt の方向がずれていないかを、テストで見張る形を強めた。

## 実OpenAI検証結果

- 実OpenAI で同一候補 1 件を検証した。
- 1 候補 1 Responses API を維持した。
- Judge API は 0 回。
- 自動再生成は 0 回。
- X API の追加取得は 0 回。
- X 投稿は 0 回。
- 本番 Firebase 書き込みは 0 回。
- 生成結果は、A/B が送信可能水準、C はやや説明調になりやすい、という評価だった。

## APIコスト方針

- 新規生成は引き続き 1 候補 1 API に統一する。
- 追加の Judge API や品質理由の自動再生成は使わない。
- Context は最大 2 件までに抑え、できるだけ 1 件で済ませる。
- X API 側は初回最大 50 件、通常最大 20 件、1 操作 1 API、5 分クールダウンの方針を維持する。
- 実API、X 投稿、本番 Firebase 書き込みは、必要な安全条件がそろうときだけ扱う。

## 今回の制約

- Git 操作はしない。
- コード変更はしない。
- 実OpenAI の追加実行はしない。
- X API、本番 Firebase、本番デプロイは使わない。
- 返信品質改善の事実確認と、RC 文書化だけを行った。
